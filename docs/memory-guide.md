# Memory Guide

## 목적
`Cycle` 의 Memory Engine V2 를 실제 workflow/task 코드에서 어떻게 써야 하는지 정리한다.

## 핵심 개념
- `ctx.memory` 는 단순 key/value store 가 아니라 `MemoryEngine` 이다.
- 모든 memory record 는 shard 와 kind 를 가진다.
- shard:
  - `user`
  - `task`
  - `workflow`
  - `system`
  - `knowledge`
- kind:
  - `raw`
  - `summary`
- 모든 task 는 `memoryPhase` 와 `memoryTaskType` 을 반드시 선언해야 한다.
- runtime 은 task 실행 직전에 `beforeStep()` retrieve 를, 직후에 `afterStep()` write 를 자동 호출한다.

## 자동 조회 흐름
task 실행 직전에 runtime 이 자동으로 아래 순서를 수행한다.

1. `task.memoryTaskType` 기반 shard routing
2. hybrid retrieval
3. top-k context assembly
4. 결과를 `ctx.memoryContext` 로 주입

즉 task 안에서 기본 조회를 직접 구현할 필요는 없다.

```ts
class GenerateSummaryTask extends Task {
  name = "generateSummary";
  memoryPhase = "EXECUTION" as const;
  memoryTaskType = "workflow" as const;
}
```

자동 조회 결과는 `ctx.memoryContext` 에 들어온다.

```ts
const retrievedContext = ctx.memoryContext?.assembledContext ?? "none";
```

관련 구현:
- [cycle.ts](../src/cycle.ts)
- [memory.ts](../src/memory.ts)

## AI 질의 연결 방식
조회 자체는 자동이지만, 조회된 context 를 `ctx.ai.chat()` 또는 `ctx.ai.chatStream()` 요청에 붙이는 것은 task 코드가 직접 한다.

```ts
const completion = await ctx.ai.chat({
  messages: [
    {
      role: "developer",
      content: "You summarize workflow inputs into a short execution brief."
    },
    {
      role: "user",
      content:
        `Summarize the following input.\n\n` +
        `${sourceText}\n\n` +
        `Retrieved context:\n${ctx.memoryContext?.assembledContext ?? "none"}`
    }
  ]
});
```

예시:
- [openai-summary-workflow.ts](../src/examples/openai-summary-workflow.ts)
- [openai-streaming-summary-workflow.ts](../src/examples/openai-streaming-summary-workflow.ts)

정리:
- 자동 수행: 조회와 `ctx.memoryContext` 주입
- task 가 직접 하는 일: AI 프롬프트에 `ctx.memoryContext` 를 포함

## 수동 조회
기본 hook 외에 task 안에서 추가 조회가 필요하면 `ctx.memory.retrieve()` 를 직접 호출하면 된다.

```ts
const retrieved = await ctx.memory.retrieve({
  query: "project workflow summary",
  taskType: "workflow",
  phase: "PLANNING"
});
```

예시:
- [report-workflow.ts](../src/examples/report-workflow.ts)

언제 쓰는가:
- 기본 `beforeStep()` 결과와 별개로 더 구체적인 query 가 필요할 때
- 한 task 안에서 서로 다른 shard routing 을 여러 번 써야 할 때
- 검색 결과 자체를 artifact 나 보고서로 남기고 싶을 때

## 명시적 write
automatic `afterStep()` write 가 있더라도, task 의미를 가진 요약/결정/산출물은 task 안에서 `ctx.memory.write()` 로 직접 저장하는 편이 좋다.

```ts
await ctx.memory.write({
  id: `memory.workflow.summary.${ctx.workflowId}.generateSummary`,
  shard: "workflow",
  kind: "summary",
  payload: {
    workflowId: ctx.workflowId,
    currentStep: "generateSummary",
    history: [],
    contextSummary: completion.outputText
  },
  description: "Summary generated by AI chat provider",
  keywords: ["summary", "ai", "chat"],
  importance: 0.92,
  workflowId: ctx.workflowId,
  runId: ctx.runId,
  sourceTask: "generateSummary",
  phase: "EXECUTION",
  taskType: "workflow"
});
```

## 어떤 shard 를 쓸지
- `user`
  - 사용자 선호, 행동 패턴, 개인화 규칙
- `task`
  - step 단위 실행 상태, 실패, 재시도, raw execution facts
- `workflow`
  - workflow 레벨 요약, 단계 진행, 설계/요구사항/결정
- `system`
  - 정책, 제약, 강제 규칙
- `knowledge`
  - runbook, 문서, RAG 입력, 외부 지식

## raw 와 summary
- `raw`
  - 원본 실행 facts, 세부 기록, input/output, repeated event source
- `summary`
  - 후속 task 가 바로 재사용할 요약본

권장 규칙:
- 후속 task 가 직접 읽을 핵심 결과는 `summary`
- 반복 실패, 디버그 단서, 원본 실행 정보는 `raw`

## routing 규칙
`memoryTaskType` 별 기본 shard routing 은 고정이다.

```ts
user -> ["user"]
workflow -> ["workflow", "task"]
debug -> ["task", "knowledge"]
default -> ["knowledge"]
```

즉 task 선언 자체가 자동 조회 범위를 결정한다.

## memoryInjection 과 rag
run 시작 전에 seed memory 를 넣고 싶으면 `memoryInjection` 과 `rag` 를 사용한다.

```ts
await cycle.run("hook-workflow", input, {
  memoryInjection: [
    {
      id: "memory.user.summary.hook-user",
      shard: "user",
      kind: "summary",
      payload: {
        userId: "hook-user",
        preferences: ["line output"],
        behaviorPatterns: ["reviews context before execution"],
        lastUpdated: Date.now()
      },
      description: "Hook user memory",
      keywords: ["user", "hook"],
      importance: 0.92,
      phase: "PLANNING",
      taskType: "user",
      sourceTask: "seed.user"
    }
  ],
  rag: [
    {
      id: "hook-runbook",
      text: "Knowledge runbook for hook retrieval",
      meta: {
        keywords: ["knowledge", "hook", "runbook"]
      }
    }
  ]
});
```

- `memoryInjection` 은 원하는 `MemoryRecordInput[]` 를 직접 넣는다.
- `rag` 는 runtime 이 자동으로 `knowledge/raw` record 로 변환한다.

예시:
- [cycle.test.ts](../tests/cycle.test.ts)

## lifecycle
Memory Engine 은 아래 lifecycle 을 적용한다.

```text
Create -> Update -> Merge -> Compress -> Expire
```

현재 구현 기준:
- importance `< 0.6` 이면 discard
- 같은 id 는 overwrite
- 유사 record 는 merge
- 반복 task raw event 는 compress
- 오래된 record 는 archive
- importance `< 0.3` 이면 delete

검증 예시:
- [memory.test.ts](../tests/memory.test.ts)
- [memory-demo-workflow.ts](https://github.com/skyend/agentic-task-kit-axpm/blob/main/example-project/src/memory-demo-workflow.ts)

## Ink 에서 보는 방법
`CYCLE_RENDER_MODE=ink` 로 실행하면 우측 패널에서 아래 메모리 이벤트를 볼 수 있다.

- `memory.before_step`
- `retrieval.performed`
- `memory.write`
- `memory.merge`
- `memory.compress`
- `memory.archive`
- `memory.expire`

예시:

```bash
CYCLE_RENDER_MODE=ink npm run example
```

AXPM demo:

```bash
git clone https://github.com/skyend/agentic-task-kit-axpm.git
cd agentic-task-kit-axpm/example-project
npm run start:memory-demo:ink
```

## 권장 패턴
- task 는 항상 `memoryPhase`, `memoryTaskType` 을 먼저 선언한다.
- 자동 조회 결과는 `ctx.memoryContext` 로 받고, AI task 는 그 내용을 프롬프트에 직접 포함한다.
- 장기적으로 재사용할 산출물만 `summary` 로 직접 저장한다.
- 디버그성 실행 흔적은 `task/raw` 에 남기고 lifecycle/compression 에 맡긴다.
- run 시작 seed data 는 `memoryInjection` / `rag` 로 넣는다.

## 흔한 실수
- `ctx.memoryContext` 가 자동으로 AI provider 요청에 붙는다고 가정하는 것
- `memoryPhase` / `memoryTaskType` 선언 없이 task 를 만드는 것
- 모든 것을 `workflow/summary` 로만 저장하는 것
- 후속 task 가 읽을 데이터인데 `raw` 로만 남기고 별도 summary 를 쓰지 않는 것
- task 안에서 `console.log` 로 디버깅하고 memory/log panel 을 활용하지 않는 것
